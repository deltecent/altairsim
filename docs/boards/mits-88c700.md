# MITS 88-C700 — Centronics Printer Controller

**Status:** implemented, `type = "c700"` — **polled** (see *Limitations*)

## The real hardware

The S-100 interface card that drove the **Altair C700 line printer** (MITS, first printing
March 1977) — a bidirectional 5×7 dot-matrix printer, a 64-character USASCII subset, 132
columns at 10 cpi. The card is a Centronics-parallel controller: the guest writes a byte to a
data port, a strobe clocks it into the printer, and a status port reports whether the printer
will take the next one.

It is a plain output card. Where the [88-SIO](mits-88sio.md) wraps a UART with a receiver, a
transmit deadline and a word frame, the C700 has **no chip and no receive path** — a printer
sends nothing back. Configuration is a pair of DIP switches (SW2/SW3) selecting the base
address; the MITS default, which MITS software requires, is **002 octal**.

Two consecutive ports, an even/odd pair like the 88-SIO: **Control/Status at the even base**,
**Data at the odd address above it**. Address bit A0 picks the channel, so the base is always
even.

## Register reference

Two ports. Control/Status at an **even** base, Data at base+1.

| Addr | OUT (write) | IN (read) |
|---|---|---|
| BASE+0 | **Control.** D0 = PRIME (**active low**: 0 resets the buffer, homes the head), D1 = interrupt enable. D2–D7 ignored. | Status word (below) |
| BASE+1 | Data — a character to the printer | *(write-only; reads float to `0xFF`)* |

### The status word (reference Table 1) — **ACTIVE HIGH**

Unlike the 88-SIO's *inverted* ready bits, the C700's status reads true-sense: a set bit means
the named condition **is** so. Bit 5 is unused.

| bit | name | HIGH means |
|---|---|---|
| 0 | ACKNOWLEDGE | the printer will accept a new byte |
| 1 | BUSY | print / carriage-return / line-feed in progress |
| 2 | PAPER EMPTY | out of paper |
| 3 | SELECT | **not** selected (select is active-low: 0 = selected) |
| 4 | FAULT | paper out or not selected |
| 6 | INTERRUPT ENABLE | interrupts are enabled |
| 7 | INTERRUPT REQUEST | an interrupt is being requested |

### The control word (reference Table 2)

| bit | name | effect |
|---|---|---|
| 0 | PRIME | **LOW** resets the printer's buffer counter and homes the head; HIGH does nothing |
| 1 | INTERRUPT CONTROL | HIGH enables the interrupt structure, LOW disables it |

### Printer control codes

These are interpreted by the **printer**, not the card — they are ordinary data bytes on the
line. LF (0x0A) advances a line; CR (0x0D) prints the buffer then line-feeds; DC1/DC3
(0x11/0x13) select/deselect; DEL (0x7F) clears the buffer and homes the carriage; SO (0x0E)
prints the buffer double-width. The card passes them through untouched.

## How it is simulated

**A bare latch, not a chip** (`src/boards/mits-88c700.{h,cpp}`). The board owns a single
`ByteStream` and the port decode around it — no `Uart1602`, because there is nothing to
receive and no character-time frame to model. Writing the data port is the whole of the card's
job; reading the status port reflects whether the line will take a byte.

- **Decodes** `IoWrite` at `BASE` and `BASE+1`, but `IoRead` **at `BASE` only**. No memory.
  The direction is part of the decode because the two channels face different ways: there is
  no `IN` at the odd address anywhere in the manual, a printer sends nothing back up the
  cable, and a real card's status buffer is gated on A0=0 — so on `IN BASE+1` nothing turns
  on and the card leaves the bus alone. The read then floats to `0xFF` **from the bus**,
  which is the only thing allowed to say it (DESIGN.md §4.6.1). `port` **must be even** —
  the decode ignores A0 and uses it to select the channel, so an odd base is refused with a
  sentence saying why.
- **The status** is derived from the stream: `ACKNOWLEDGE` (ready) tracks `writable()`, and its
  complement lights `BUSY`. A file, a socket or the console is a buffered sink that is
  essentially always ready — the honest analogue of a printer that is keeping up. `PAPER EMPTY`,
  `SELECT` and `FAULT` are always clear: a file cannot run out of paper. The bits exist so that
  the day a real parallel port reports one, it lands where the manual says.
- **PRIME** (control D0 low) resets the printer's buffer on real hardware. This card holds no
  buffer — bytes go straight out — so PRIME **flushes** the line, which is the honest
  equivalent, and the guest may issue it harmlessly at any time.
- **The line is raw.** The bytes the guest sends are the bytes on the wire, control codes and
  all. There is **no transform chain** — that belongs to the console alone (DESIGN.md §7.2). A
  captured file is a byte-for-byte record of what the printer received; it is **not** reformatted
  into a printed page (the card is a controller, not a print mechanism).
- **`pump()`** forwards to the stream, so a `socket:` endpoint accepts and drains and a `file:`
  capture is flushed while the machine runs, not only at `DISCONNECT`.
- **Where the bytes go is the operator's `CONNECT`**, not the card's business (DESIGN.md §7.7):
  a `file:`, the `console`, a `socket:`, a real `serial:` printer, or `null`. The card knows it
  has a line; it does not know what is on the end of it.
- Does **not** master the bus, and (see below) asserts no interrupt.

### The `file:` endpoint

`CONNECT lpt0:prn file:printout.txt` captures the printout to a host file
(`src/host/file.{h,cpp}`). It is a generic, write-only sink — any card can use it, not just the
printer — opened **binary and truncating**, so it is 8-bit clean and a fresh file each connect.
Its `describe()` returns the exact `file:PATH` spec, so `CONFIG SAVE` round-trips it. A relative
path follows the usual rule (config-relative in a machine file, cwd-relative typed at the
prompt); the board rebases it through `resolvePath()` and remembers the path *as written*, so a
reload does not double-rebase.

## Limitations and deliberate departures

**The interrupt structure is not modeled — the card is polled.** The real 88-C700 has a
single-level interrupt, switchable to fire after each character or after each CR/LF (SW2 #4).
Here, the INTERRUPT ENABLE bit (control D1) is stored and read back at status bit 6, but **no
request is raised and no wire is pulled**: status bit 7 (INTERRUPT REQUEST) stays clear and the
card asserts nothing on pin 73. A polled driver — write a byte, poll ACKNOWLEDGE/BUSY — is
complete and is how print routines of the era typically worked. Interrupt support is a
deliberately separate addition, deferred pending a use for it (issue #26).

**PAPER EMPTY, SELECT and FAULT are always "OK".** A `ByteStream` sink cannot run out of paper
or deselect. Synthesizing those states would invent a hardware event the transport does not
have. If a real parallel-port endpoint ever lands, they are genuine events and can be reported
from the place that knows.

**No page rendering.** The capture is the raw byte stream, so its line endings are whatever the
program emitted (often a bare CR, since the physical printer supplied the line feed). Turning
that into host-native text would be modeling the *printer mechanism*, which this card is not —
and it would be a transform on the line, which the design forbids. Normalize with an editor if
you want DOS/Unix line breaks.

## Verification

`tests/test_c700.cpp`, with `Bus::setVerify(true)` on:

1. **The card** — one unit `prn`, two ports and not a third, no memory, and an **odd base is
   refused with a reason**; the decode follows the base when it moves.
2. **Status is active-high** — an idle byte-sink reads ready (ACKNOWLEDGE set, BUSY clear), and
   the write-only data channel reads `0xFF` — checked for **provenance** as well as value: the
   card must not claim the read, so the `0xFF` is the bus's (issue #26).
3. **A character written to the data port lands on the line**, in order.
4. **The line is raw** — CR/LF/SO/DEL pass through byte-for-byte.
5. **The control channel** — D1 arms/disarms the interrupt-enable bit (status bit 6); PRIME
   (D0 low) does not eat characters already sent; enabling interrupts raises **no** request and
   pulls **no** wire (the polled-card contract).
6. **Disconnect leaves a dead line**, not a dangling pointer, and names the one real unit.
7. **`connect` round-trips** the endpoint spec through the property (for `CONFIG SAVE`).
8. **The `file:` endpoint** — resolves, describes itself exactly, is write-only, survives
   control bytes byte-for-byte, and refuses an unopenable path with a reason.
