# 88-2SIO — MITS Dual Serial Interface

**Status:** milestone 1. **The proof vehicle** — a fully-modeled 2SIO exercises every interface in the design (console, TCP socket, host serial, interrupts, multi-unit boards, multiple instances), which is why it is the only peripheral in milestone 1.

## The real hardware

Two independent **Motorola 6850 ACIA**s on one S-100 card. Base address is jumper-selectable; channel A sits at BA+0/BA+1 and channel B at BA+2/BA+3. Baud rate is a jumper/clock on the board, not software-settable. Each ACIA's interrupt request can be jumpered to the S-100 interrupt lines.

Default base **0x10**, so: channel A = 0x10 (control write / status read) + 0x11 (data), channel B = 0x12 + 0x13.

## Register reference

### Control register (write) — the Python prototype ignored this entirely; don't.

| Bits | Meaning |
|---|---|
| 0–1 | Clock divide: `00` = ÷1, `01` = ÷16, `10` = ÷64, **`11` = master reset** |
| 2–4 | Word select: data bits / parity / stop bits (8 standard combinations) |
| 5–6 | Transmit control: `00` RTS low + **TIE off**; `01` RTS low + **TIE on** (transmit interrupt enable); `10` RTS high + TIE off; `11` RTS low + transmit break |
| 7 | **RIE** — receive interrupt enable |

Period software writes `0x03` (master reset) then `0x11` (÷16, 8N2). An **interrupt-driven** driver sets bit 7 and/or the `01` transmit-control field — that is the path milestone 1 must prove.

### Status register (read) — **true sense**

| Bit | Name |
|---|---|
| 0 | **RDRF** — receive data register full |
| 1 | **TDRE** — transmit data register empty |
| 2 | DCD |
| 3 | CTS |
| 4 | FE |
| 5 | OVRN |
| 6 | PE |
| 7 | **IRQ** — this ACIA's interrupt request |

> **Trap:** the 88-SIO's status bits are **inverted**, the 88-2SIO's are not. A machine with both boards has both conventions live at once. Do not share code between them without thinking.

## How it is simulated

- Decodes `IoIn`/`IoOut` on BA+0 … BA+3.
- **Two units** (`a`, `b`), each an independent 6850 with its own `ByteStream` and its own properties. `CONNECT sio2a:a console`, `CONNECT sio2a:b socket:2323`.
- **Multiple board instances** at different base ports: `sio2a` at 0x10, `sio2b` at 0x14.
- Bit 7 (IRQ) reflects the ACIA's own request; the board's `interrupt` property says where that request is jumpered — `none | int | vi0..vi7`. With `interrupt = int` and no VI board present, the `IntAck` cycle finds nobody driving the bus, the CPU reads a floating `0xFF`, and executes **RST 7**. That is the real Altair behavior.
- **TDRE is driven from the `EventQueue` at the configured baud rate** — see below.

### Properties

| Property | Scope | Runtime? | Notes |
|---|---|---|---|
| `port` | board | config | Base address |
| `interrupt` | board | config | `none \| int \| vi0..vi7` |
| `baud` | per unit | **yes** | The board-specific setting through the generic interface: `SET sio2a:a BAUD=9600` |
| `connect` | per unit | yes | Endpoint (`CONNECT` sets it) |

### Reset

- `Reset::PowerOn` and `Reset::Bus` both: 6850 master reset — clear RDRF/TDRE, drop RTS.
- **Both keep the `ByteStream` connected.** A warm reset does not unplug the terminal.

## Quirks reproduced (and what breaks if you don't)

**Model TDRE timing properly.** The Python prototype hardwired `TDRE = 1` (always ready to send). That is not merely an approximation:

> The Mike Douglas Altair CP/M BIOS **infers the line speed by timing how long TDRE stays clear**. It counts ~21 µs ticks while polling; if the 16-bit counter's high byte reaches ≥ 11, it concludes **110 baud Teletype** and sets `sndNull`, so a NULL is sent after every CR.

Hardwire TDRE and you **silently change what the guest does**. Drive TDRE from the `EventQueue` at the configured baud rate.

**Honor the control register.** Master reset (`0x03`) must actually reset. The interrupt-enable bits must actually enable interrupts — that is the whole point of milestone 1's acceptance test 4.

## Limitations

- Baud is a board jumper on real hardware, so `baud` is a property rather than a register. Software cannot change it, which is correct.
- DCD and CTS are driven from the `ByteStream`'s `Status()`; for a console or file endpoint they are simply asserted.
- Parity and framing errors are not synthesized (no line noise model). OVRN *is* modeled — a guest that ignores RDRF long enough should see it.

## Verification (milestone 1 acceptance)

1. 4K MITS BASIC `LOAD`ed into RAM answers `PRINT 2+2` on the console, and over MCP via `send`/`expect`.
2. Same session over a **TCP socket** with a terminal emulator attached.
3. Same over a **host serial port**, on macOS/Linux and Windows.
4. An **interrupt-driven** console echo program (not polled) works with **no VI board in the machine** — proving RIE/TIE → `pINT` → `IntAck` → floating-bus RST 7.
5. **Two 2SIO boards at once**, four channels live, each independently configured.
